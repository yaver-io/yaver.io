# Yaver Deploy Runbook

Explicit, copy-paste-ready instructions for every Yaver deploy target.
Intended to be followed by a human, by Claude Code, or by Codex — if a
command here fails, read the error carefully and consult the
**Troubleshooting** section at the bottom before improvising.

## Safety rules (read first)

1. **Never commit secrets.** Every credential must come from
   `.env.test`, env vars, or GitHub Actions secrets. The repo is
   public on GitHub.
2. **Never push/commit/force-push without the user's explicit OK.**
3. **Never run `--no-verify` / `--no-gpg-sign` / `git reset --hard`
   on tags or remote branches** unless explicitly told.
4. **Always run `mobile-cache-cleanup.sh preflight` before a mobile
   deploy.** It fails hard below 20 GB free disk and unclogs stale
   caches across sfmg/talos/yaver/botox.
5. **Bump `versions.json` first, then run `./scripts/sync-versions.sh`,
   then tag.** CI validators refuse to release if the tag's version
   does not match `versions.json` for that component.

## The 9 deploy targets

| Target | Trigger | Surface | Script (manual) | Workflow (CI) |
|--------|---------|---------|-----------------|----------------|
| iOS (TestFlight) | manual or `mobile/v*` tag | App Store Connect | `scripts/deploy-testflight.sh` | `.github/workflows/release-mobile.yml` |
| Android (Play Store) | manual or `mobile/v*` tag | Play Console internal track | `scripts/deploy-playstore.sh` + `scripts/upload-playstore.py` | `.github/workflows/release-mobile.yml` |
| Web (yaver.io) | `web/v*` tag | Cloudflare Workers | `scripts/deploy-web.sh` | `.github/workflows/release-web.yml` |
| Convex backend | manual | Convex prod | `cd backend && npx convex deploy --yes` | n/a |
| Relay server | `relay/v*` tag | Hetzner / self-host | `cd relay/deploy && ./up.sh <ip> --docker` | `.github/workflows/release-relay.yml` |
| CLI (`yaver`) | `cli/v*` tag | npm | see §CLI below | `.github/workflows/release-cli.yml` |
| SDKs (`yaver-sdk`, `yaver` python) | GitHub release published | npm + PyPI + pub.dev | see §SDKs below | `.github/workflows/release-sdk.yml` |
| Desktop installer | `installer/v*` tag | macOS `.dmg` + Windows `.exe` + Linux `.AppImage` | electron-builder | `.github/workflows/release-installer.yml` |
| Pi image | `piImage/v*` tag | GitHub release asset | `scripts/release-pi-image.sh` | `.github/workflows/release-pi-image.yml` |

For releases that go out to users (CLI, SDKs, mobile, web), **prefer
the CI path**. It signs, notarises, runs integration tests, bumps
package managers, and fails fast on a mismatched version. Manual
scripts are for local iteration, emergency re-uploads, or TestFlight
dogfooding.

## Prereqs: one-time environment setup

Export these in your shell (`~/.zshrc`) or in a local `.env.test` that
you `source` before running manual deploys. Do NOT commit the actual
values — the paths / IDs below are correct for this repo, values
come from the password manager.

```bash
# iOS (App Store Connect API key — lives outside this repo)
export APP_STORE_KEY_PATH="<your-apple-key-path>"
export APP_STORE_KEY_ID="<your-apple-key-id>"
export APP_STORE_KEY_ISSUER="<your-issuer-uuid>"
export APPLE_TEAM_ID="<your-apple-team-id>"

# Android — keystore + Play service account (both in keys/, gitignored)
# keystore.properties is regenerated by expo prebuild; see §Android

# Play Store upload script expects this:
export PLAY_STORE_KEY_FILE="keys/google-play-service-account.json"

# Tools
which xcodebuild     # Xcode 15+
/usr/libexec/java_home -v 17   # Java 17 for Android
which node           # Node 20+
which go             # Go 1.22+
which python3        # Python 3.10+
```

Install the mobile cache helper once (shared across all mobile
projects on this Mac):

```bash
ls -l ~/.local/bin/mobile-cache-cleanup.sh   # must exist and be +x
```

## iOS — TestFlight (manual)

One command:

```bash
mobile-cache-cleanup.sh preflight && \
./scripts/deploy-testflight.sh
```

`deploy-testflight.sh` auto-bumps `CFBundleVersion` in
`mobile/ios/Yaver/Info.plist`, archives, exports, and uploads to
App Store Connect via the API key. Successful output ends with:

```
Uploaded Yaver
** EXPORT SUCCEEDED **
✓ TestFlight build <N> uploaded
```

Then mark the deploy so stale-cache cleanup does not purge Yaver's
Xcode caches on the next `weekly` run:

```bash
mobile-cache-cleanup.sh mark-deployed yaver
```

### If the archive fails with "Build input file cannot be found: ... /RNWhisperSpec/States.cpp"

The React Native codegen step did not generate the shadow-node
`.cpp` files Xcode expects. Regenerate manually, then re-run the
script:

```bash
cd mobile
node node_modules/react-native/scripts/generate-codegen-artifacts.js \
  -p . -o ios/build/generated -t ios
cd ..
./scripts/deploy-testflight.sh
```

Verify the file exists before re-archiving:

```bash
ls mobile/ios/build/generated/ios/react/renderer/components/RNWhisperSpec/States.cpp
```

This gotcha typically appears after `npx expo prebuild --clean`, a
corrupt `Pods` state, or a node_modules install that ran without the
codegen postinstall hook.

### If Apple returns "Upload limit reached. Please wait 1 day"

TestFlight rate-limits to ~15–20 uploads per app per day. There is
no API to reset or query the remaining quota. Wait ~24 hours.
**Use direct-to-device USB install (`npx expo run:ios --device <UDID>`)
for iteration**, not TestFlight.

### Re-export an existing archive without re-building

The archive survives at `/tmp/Yaver.xcarchive` between runs. If only
the upload failed (e.g. Apple transient 500, exit 70):

```bash
xcodebuild -exportArchive \
  -archivePath /tmp/Yaver.xcarchive \
  -exportOptionsPlist /tmp/ExportOptions.plist \
  -exportPath /tmp/YaverExport -allowProvisioningUpdates \
  -authenticationKeyPath "$APP_STORE_KEY_PATH" \
  -authenticationKeyID "$APP_STORE_KEY_ID" \
  -authenticationKeyIssuerID "$APP_STORE_KEY_ISSUER"
```

## Android — Google Play (manual)

One command (builds the AAB and uploads to internal track):

```bash
mobile-cache-cleanup.sh preflight && \
JAVA_HOME=$(/usr/libexec/java_home -v 17) ./scripts/deploy-playstore.sh && \
PLAY_STORE_KEY_FILE=keys/google-play-service-account.json \
  python3 scripts/upload-playstore.py && \
mobile-cache-cleanup.sh mark-deployed yaver
```

`deploy-playstore.sh` auto-bumps `versionCode` in
`mobile/android/app/build.gradle`, runs `./gradlew bundleRelease`,
and prints the AAB path. `upload-playstore.py` finds the AAB and
uploads it to the **internal testing** track via the Google Play
Publishing API using the service account at
`keys/google-play-service-account.json`.

Successful output ends with:

```
Created edit: <id>
Uploaded bundle: versionCode=<N>
Assigned to internal track
Edit committed! Build <N> is live on internal track.
```

### Keystore + service account (expected layout)

```
keys/
  yaver-upload.keystore           # upload keystore (gitignored)
  google-play-service-account.json # Play service account (gitignored)
```

`mobile/android/keystore.properties` — **regenerated after every
`npx expo prebuild --platform android --clean`**. It is gitignored
and must be restored from the password manager afterward:

```properties
storeFile=../../../keys/yaver-upload.keystore
storePassword=<from password manager>
keyAlias=yaver-upload
keyPassword=<from password manager>
```

`mobile/android/app/build.gradle` must contain the
`signingConfigs.release` block that reads this file — if a clean
prebuild wipes it, releases silently sign with the debug keystore
and Play rejects with a SHA1 mismatch.

### If Play rejects with SHA1 mismatch

The upload keystore's SHA1 does not match the SHA1 Play Console
expects for this app. Two options:

1. **Request upload key reset in Play Console** → Settings → App
   integrity → Upload key → *Request upload key reset*. Takes
   24–48h. Upload the public cert of the new keystore afterward.
2. Find the original keystore on another machine.

### Common compile warning (not a failure)

`expo-modules-core` on Expo SDK 52 logs
`Could not get unknown property 'release' for SoftwareComponent`.
If it turns into a failure, patch
`node_modules/expo-modules-core/android/ExpoModulesCorePlugin.gradle`
line ~91: wrap `from components.release` in
`if (components.findByName('release') != null)`.

## Web (yaver.io — Cloudflare Workers)

Manual deploy (useful for testing from a dev branch):

```bash
./scripts/deploy-web.sh
```

The script name is kept for backward compat; it builds with
`@opennextjs/cloudflare` and deploys via `wrangler`. It enforces a
**10 MB cap** on `web/` (excluding `node_modules`, `.next`,
`.open-next`). Current weight is ~2.5 MB.

Or, invoke the underlying commands directly:

```bash
cd web
npm ci
npm run deploy   # runs opennextjs-cloudflare && wrangler deploy
```

Environment:
- Secrets live in Cloudflare (`wrangler secret put <KEY>`), not in
  `wrangler.toml`.
- Routes are configured in `web/wrangler.toml`.
- DNS for `yaver.io` is in the Cloudflare dashboard.

**CI path:** push a tag like `web/v1.1.11` after bumping
`versions.json` → `release-web.yml` runs.

GitHub secrets needed: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`,
`CONVEX_DEPLOY_KEY`.

## Relay server

Self-hostable Go QUIC relay. Two paths:

### Docker (recommended)

```bash
cd relay/deploy
./up.sh <server-ip> --docker
# stop: ./down.sh <server-ip>
# health: curl https://<your-relay-domain>/health
```

Set the shared password via `RELAY_PASSWORD`. Agents and mobile
need it to register / proxy through the relay.

### Binary (no Docker)

```bash
cd relay/deploy
./up.sh <server-ip> --binary
```

### CI release

Push `relay/v0.1.1` → `release-relay.yml` builds the Go binary for
linux/arm64 + linux/amd64 and publishes a GitHub release.

## CLI (`yaver`) — full fan-out release

The CLI ships to **nine** package managers from a single tag.
Everything runs in GitHub Actions when you push a `cli/v*` tag.
Local manual builds are for testing only — **do not push tagged
binaries by hand**.

### Triggering the release

```bash
# 1. Bump
jq '.cli = "1.96.10"' versions.json > /tmp/v.json && mv /tmp/v.json versions.json
./scripts/sync-versions.sh            # propagates to cli/package.json, main.go, etc.

# 2. Commit + tag
git add versions.json cli/package.json desktop/agent/main.go
git commit -m "Bump cli to v1.96.10"
git tag cli/v1.96.10
git push github main
git push github cli/v1.96.10          # triggers release-cli.yml

# 3. Watch
gh run watch
```

### What the workflow does (`release-cli.yml`)

1. **validate** — tag must match `versions.json.cli`.
2. **publish-npm** — `npm publish --access public` for
   `cli/` as `yaver-cli`.
3. **publish-mcp-registry** — pushes `server.json` to the MCP
   Registry (OIDC, no stored secret).
4. **build** (matrix) — cross-compiles for
   `darwin-arm64`, `darwin-amd64`, `linux-arm64`, `linux-amd64`,
   `windows-amd64`. Produces: `.tar.gz`, `.dmg`, `.exe`, `.zip`,
   `.deb`, `.rpm`. Uploads as workflow artifacts.
5. **release** — creates a **draft** GitHub release with all
   artifacts + `checksums.txt`. Windows `.exe` is unsigned at this
   point; sign with SimplySign Desktop before publishing the draft.
6. **docker-build** + **docker-manifest** — builds multi-arch
   (amd64 + arm64) images and pushes `kivanccakmak/yaver-cli:latest`
   + `:<version>` to Docker Hub and `ghcr.io`.

### User install commands after release

```bash
npm install -g yaver-cli

# Docker
docker run --rm -it ghcr.io/kivanccakmak/yaver.io/cli:latest
```

### Local Go build (dev only, do NOT push)

```bash
cd desktop/agent
GOOS=darwin GOARCH=arm64 go build -ldflags="-s -w" -o yaver-darwin-arm64 .
GOOS=darwin GOARCH=amd64 go build -ldflags="-s -w" -o yaver-darwin-amd64 .
GOOS=linux  GOARCH=arm64 go build -ldflags="-s -w" -o yaver-linux-arm64  .
GOOS=linux  GOARCH=amd64 go build -ldflags="-s -w" -o yaver-linux-amd64  .
GOOS=windows GOARCH=amd64 go build -ldflags="-s -w" -o yaver-windows-amd64.exe .
```

### Windows code signing (only needed if you manually upload `.exe`)

Requires SimplySign Desktop logged in:

```bash
./keys/sign-windows.sh yaver-windows-amd64.exe
```

## SDKs — `yaver-sdk` (npm) + `yaver` (PyPI) + Flutter pub.dev

The SDK release pipeline triggers **on GitHub release publish**, so
the flow is: create a tagged release (typically the CLI release),
publish the draft, and the SDK workflow fires.

### `release-sdk.yml` publishes

1. **npm** (`sdk/js/`) — `npm publish --access public` using
   `NPM_TOKEN`.
2. **PyPI** (`sdk/python/`) — `python -m build` then
   `twine upload dist/*` with `TWINE_USERNAME=__token__`,
   `TWINE_PASSWORD=$PYPI_TOKEN`.
3. **C shared library** — per-platform Go build-mode=c-shared
   artifacts attached to the release.

### Flutter / pub.dev (manual — no CI)

Requires a pub.dev account linked to a verified publisher.

```bash
cd sdk/flutter
dart pub publish        # --dry-run first to preview
```

### Go SDK

No publish step — Go modules import directly from GitHub via
`go get github.com/kivanccakmak/yaver.io/sdk/go/yaver`.

### Manual npm/PyPI publish (if CI is down)

```bash
# yaver-cli (push-to-device CLI — different package from sdk/js!)
echo "//registry.npmjs.org/:_authToken=$NPM_TOKEN" > cli/.npmrc
cd cli && npm ci && npm publish --access public
rm cli/.npmrc   # do NOT commit

# yaver-sdk (programmatic SDK)
echo "//registry.npmjs.org/:_authToken=$NPM_TOKEN" > sdk/js/.npmrc
cd sdk/js && npm ci && npx tsc && npm publish --access public
rm sdk/js/.npmrc

# yaver (Python)
cd sdk/python
python -m build
python -m twine upload dist/* --username __token__ --password "$PYPI_TOKEN"
```

Before publishing `yaver-cli`:
- `cli/sdk-manifest.json` matches `mobile/sdk-manifest.json`
- `cli/hermesc/` contains the hermesc binaries for all
  platforms
- `cli/package.json`'s version is bumped in lockstep with
  `versions.json.cli`

## Version management

The single source of truth is `versions.json` at the repo root.
Bump the component you want to release, then propagate:

```bash
# Bump (example: cli)
node -e "
  const fs = require('fs'); const v = require('./versions.json');
  v.cli = '1.96.10';
  fs.writeFileSync('./versions.json', JSON.stringify(v, null, 2) + '\n');
"

# Propagate to every downstream file
./scripts/sync-versions.sh
```

`sync-versions.sh` updates:
- `cli/package.json`
- `desktop/agent/main.go` (`const version`)
- `mobile/app.json`, `mobile/ios/Yaver/Info.plist`,
  `mobile/ios/Yaver.xcodeproj/project.pbxproj`,
  `mobile/android/app/build.gradle`
- `web/app/page.tsx` and any other landing-page badges
- `desktop/installer/package.json`
- `relay/main.go`, `backend/package.json`, etc.

Then `scripts/check-version-bump.sh` is what CI uses to reject PRs
that modify a component but forgot to bump its version.

Tag format (enforced by each release workflow's `validate` job):

| Component | Tag | `versions.json` key |
|-----------|-----|---------------------|
| CLI | `cli/v<X.Y.Z>` | `cli` |
| Mobile | `mobile/v<X.Y.Z>` | `mobile` |
| Web | `web/v<X.Y.Z>` | `web` |
| Installer | `installer/v<X.Y.Z>` | `installer` |
| Relay | `relay/v<X.Y.Z>` | `relay` |
| Backend | `backend/v<X.Y.Z>` | `backend` |
| Pi image | `piImage/v<X.Y.Z>` | `piImage` |

## Troubleshooting

### `mobile-cache-cleanup.sh preflight` fails with < 20 GB free

Do not proceed. Free space first:

```bash
mobile-cache-cleanup.sh check        # see what's using space
mobile-cache-cleanup.sh clean-system # Xcode DerivedData + sims + Gradle
mobile-cache-cleanup.sh clean-stale 7  # purge projects idle > 7 days
# If still tight: ~/Library/Caches, ~/Downloads, old Docker images
docker system prune -a
```

### `(8 failures)` / `(2 failures)` at the end of a clean xcodebuild

The script's `tail -3` on xcodebuild output hides the real error.
Re-archive with full output:

```bash
cd mobile/ios
rm -rf /tmp/Yaver.xcarchive
xcodebuild -workspace Yaver.xcworkspace -scheme Yaver -configuration Release \
  -archivePath /tmp/Yaver.xcarchive archive \
  DEVELOPMENT_TEAM="$APPLE_TEAM_ID" CODE_SIGN_STYLE=Automatic \
  ENABLE_USER_SCRIPT_SANDBOXING=NO -allowProvisioningUpdates \
  -authenticationKeyPath "$APP_STORE_KEY_PATH" \
  -authenticationKeyID "$APP_STORE_KEY_ID" \
  -authenticationKeyIssuerID "$APP_STORE_KEY_ISSUER" \
  -derivedDataPath /tmp/YaverBuild > /tmp/ios-archive-full.log 2>&1

grep -nE "^error:|error: |fatal error:|The following build commands failed" \
  /tmp/ios-archive-full.log | head -20
```

### `Build input file cannot be found: .../RNWhisperSpec/States.cpp`

See §"If the archive fails with …" in the iOS section. Regenerate
RN codegen artifacts and re-archive.

### Android: `Could not get unknown property 'release' for SoftwareComponent`

Patch `node_modules/expo-modules-core/android/ExpoModulesCorePlugin.gradle`
line ~91 as described in §Android. This is an Expo SDK 52 / AGP
compatibility bug.

### Android: `expo-share-intent` version error

Must stay on `expo-share-intent@3.2.3` with Expo SDK 52. v4+
requires Expo 53+, v6+ requires Expo 55+.

### TestFlight: `Upload limit reached. Please wait 1 day.`

There is no API to reset. Wait 24h. Don't try to "upload just one
more" — that counts against the next day's quota too. Use
`npx expo run:ios --device <UDID>` for iteration.

### `wrangler deploy` fails on size

`web/` exceeded 10 MB. `scripts/deploy-web.sh` will print the
over-cap size. Before pushing a big asset, compress it:

```bash
ffmpeg -i input.mp4 -vcodec libx264 -crf 32 -preset veryslow \
  -vf "scale=720:-2" -an output.mp4
```

Or host externally (GitHub releases CDN, YouTube embed).

### A tag was pushed with the wrong `versions.json`

The workflow's `validate` job fails. Do NOT force-retry. Fix:

```bash
git tag -d <wrong-tag>                    # delete locally
git push github :refs/tags/<wrong-tag>    # delete remotely (OK to do for a failed validate tag)
# Fix versions.json + sync + commit, then re-tag:
git tag <correct-tag> && git push github <correct-tag>
```

### Convex mutations don't show up on the live web app

`web/` deploy and `backend/` deploy are independent. If a new web
build expects a new mutation, deploy Convex **first**:

```bash
cd backend && npx convex deploy --yes
# then
./scripts/deploy-web.sh
```

## After every deploy

1. **Mark the project** (mobile only) so cache cleanup skips it:
   ```bash
   mobile-cache-cleanup.sh mark-deployed yaver
   ```
2. **Verify in the dashboard**:
   - iOS → App Store Connect → TestFlight → Builds → latest build
     shows as "Ready to Test" or "Processing"
   - Android → Play Console → Internal testing → Releases → new
     versionCode listed
   - Web → https://yaver.io loads with the new version string
   - Convex → dashboard shows the new deployment id
3. **Smoke-test** the surface you just shipped. For mobile, pull
   the app on a real device (or TestFlight) and sign in.
4. If this was a tagged CI release: check
   `gh run list --workflow=release-<target>.yml --limit 3` to
   confirm success, and the draft GitHub release looks right
   before publishing.
