# Yaver Mesh — Interop & Robustness Audit

Date: 2026-07-20
Scope: why Yaver Mesh is not Tailscale-grade, and what it must do to coexist
with Tailscale, other VPNs, or nothing — on every device, in every combination.

Supersedes nothing. `CONNECTIVITY_MESH_AUDIT.md` (2026-07-19) found five faults
in the *connect ladder*; this document is about the **overlay itself** and the
**interop matrix**, and it corrects two claims that document left standing.

---

## 0. The one-paragraph version

Yaver Mesh has never carried a packet on this fleet. It refuses to start
whenever Tailscale is running — correctly, because its address range sits
*inside* Tailscale's — and every machine here runs Tailscale. With mesh out,
the phone has exactly two ways to reach a box: guess a LAN address, or use one
relay. On iOS the LAN guesses fail instantly for a reason the app misreports as
"no route", and the single relay is both a SPOF and something the client can
get itself **banned from**: a stale password plus five concurrent reconnect
loops trips the relay's own rate limiter, after which correct credentials are
refused too. The user experiences "Yaver Mesh has connection issues"; the truth
is that mesh was never in the path, the direct path is misdiagnosed, and the
relay outage is self-inflicted.

**The three lessons, stated once:**

1. **A default-on overlay must not choose a range inside the incumbent's.**
   Everything downstream — the standing-down, the misclassification, the
   dead data plane — follows from `100.96.0.0/12 ⊂ 100.64.0.0/10`.
2. **A failure you cannot name, you will misdiagnose.** `Network request
   failed` is RN's catch-all; treating it as "unroutable" turned a *permission
   denial* into a *routing verdict*, and then cached that verdict.
3. **Retry without coordination is an attack on yourself.** N loops × 1 relay
   × a bad credential = a rate-limit ban that outlasts the original fault.

## 1. Ground truth — measured, not inferred

Run on 2026-07-20 against the live fleet.

### 1.1 Mesh is off on every machine, and one of them says why

```
$ ssh mac-mini 'yaver mesh status'
Yaver Mesh status
  state  : off — default-on tried and could not start
  reason : interface utun4 (100.89.155.25) shares the address range Yaver Mesh
           uses (100.64.0.0/10 overlaps 100.96.0.0/12). Yaver Mesh stays off so
           it does not fight an existing VPN for routes; connections use your
           LAN or relay instead.

$ yaver mesh status            # MacBook Air
  state  : off (default-on has not completed yet — it retries on a backoff)
```

Both machines run Tailscale (`utun4` = 100.89.155.25, `utun6` = 100.89.8.111).
Mesh is therefore off **fleet-wide**, and has been since default-on shipped.

The MacBook's wording is its own defect: *"has not completed yet — it retries on
a backoff"* is indistinguishable from *"will never start while Tailscale is
up."* A status that says **wait** when the honest answer is **never, under this
configuration** is why nobody noticed for weeks.

### 1.2 What the connect logs prove

Four sessions of client logs, 08:48–12:29. The signal is in the *timings*, not
the words.

| Leg | Latency to failure | Reported as |
|---|---|---|
| `lan-tailscale 100.x` | **same second** | `Network request failed` → `unroutable` |
| `lan-heartbeat 192.168.x` | ~2 s | `timed out (no answer within the probe budget)` |

A 2 s failure is a network timeout. A **same-second** failure is the OS
refusing the socket before any I/O — the signature of the iOS Local Network
permission / ATS boundary, not of a missing route. Every tailnet leg in every
session failed instantly, including `100.89.155.25`, which is reachable from a
laptop on the same tailnet at that moment.

Other facts the logs establish:

- **Relay flaps hard.** `public-free reachable` at 08:49:37; `unreachable —
  timed out` continuously 08:52:14→08:52:54 and 18:40:32→18:43:31; healthy
  again 19:07. One relay, `relayCount: 1`, in every log line.
- **The client bans itself.**
  `09:16:45 too many invalid relay password attempts` →
  `09:19:09 invalid relay password`. The relay rate-limited the device for
  repeated bad-password attempts generated by the client's own retry storm.
- **The self-heal works, but fires too late.**
  `12:27:53 [connect-relay-auth-refresh] Refreshing relay credentials` →
  `12:27:56 [connect-success] Connected via relay`. Three seconds. It is not on
  the storm path.
- **Five concurrent reconnect loops.** At 19:07–19:09, `10.0.0.40`,
  `10.0.0.45`, `192.168.111.38`, `100.75.123.78` and `192.168.1.103` each run
  an independent backoff, each racing candidates, all against one relay host.
- **Dead candidates never age out.** `172.17.0.1` is a Docker bridge;
  `172.20.10.4/.9` an old iPhone-hotspot range; `192.168.1.105`,
  `192.168.111.8/.11` long gone. Still raced at 19:09, still timing out.
- **The UI lies while this happens.** `09:02:31 [connect-resume] Already
  connected … mode: "relay"` — nine seconds after `all 1 relay(s) failed — no
  path left to this device`.

## 2. The causal chain

The outage is not one bug. It is four, in series, each of which alone would be
survivable.

```
  Mesh range ⊂ Tailscale range
        └─► mesh stands down on every Tailscale host          [§3.1]
              └─► no overlay data plane at all
                    └─► only two paths remain: LAN guess, relay

  iOS Local Network permission / ATS boundary
        └─► every LAN + tailnet dial fails in 0 s             [§4.2]
              └─► classified "unroutable", negative-cached 5 min
                    └─► direct path removed for 5 minutes at a time

  One relay, no regions, no alternates                        [§3.3]
        └─► any relay hiccup = total outage

  Stale relay password + N uncoordinated retry loops          [§4.1]
        └─► relay rate-limits the device
              └─► correct credentials now refused
                    └─► outage outlives the original fault
```

Read bottom-up, that is exactly the user's experience: *"with Tailscale on or
off, Yaver Mesh always has connection issues."* Mesh is never in the path; the
fallbacks are individually broken; and the last one punishes retrying.

## 3. Data plane — honest gap list vs Tailscale

Yaver Mesh **is** real userspace WireGuard on a real TUN
(`desktop/agent/mesh/device.go:105`, `golang.zx2c4.com/wireguard`). The
cryptography is genuinely at parity: Curve25519 per-peer keys, private half
vault-only (`mesh/keys.go:26`), so the datapath inherits WireGuard's full
Noise_IKpsk2 handshake. What is missing is everything *around* the tunnel.

### 3.1 The range choice — the root architectural error

```
Tailscale  100.64.0.0/10 = 100.64.0.0 .. 100.127.255.255
Yaver Mesh 100.96.0.0/12 = 100.96.0.0 .. 100.111.255.255      ⊂ the above
```

`mesh/device.go:34-43` documents this and `:53-55` explicitly declines to fix
it ("fleet-breaking"). Consequences, all live:

- **Bring-up**: Tailscale installs one route for the whole `/10`, so the `/12`
  cannot be claimed without fighting it. Mesh refuses to start
  (`mesh/subnet_conflict.go:25`). Correct call, catastrophic reach: **every
  Tailscale user loses mesh entirely** — and Tailscale users are the exact
  install base a Tailscale-alternative addresses.
- **Start-order hazard**: the guard runs *only at bring-up*, and by interface
  enumeration, not route inspection. Start Tailscale **after** mesh and nothing
  reacts. Longest-prefix match then gives `/12` to mesh, so Yaver **silently
  blackholes any Tailscale peer whose address landed in 100.96–100.111** —
  roughly a 1-in-4 chance per peer.
- **Client misclassification**: `quic.ts:6136` `isMeshOverlayIP` claims
  100.96/12 first, so a *real Tailscale* address in that window is tagged
  `lan-mesh` and then skipped, because the mesh data plane isn't shipped on
  mobile. A working address is discarded as an unshipped one.

### 3.2 NAT traversal — STUN in name only

| Technique | Tailscale | Yaver |
|---|---|---|
| STUN binding | ✅ continuous, per-region | ⚠️ 3 hardcoded servers, IP only |
| Learns mapped **port** | ✅ | ❌ **discarded** |
| Candidate set per peer | ✅ many | ❌ **exactly one** |
| Connectivity checks / disco | ✅ | ❌ none |
| Hole punching, port prediction | ✅ | ❌ none |
| UPnP / NAT-PMP / PCP | ✅ | ❌ none |
| IPv6 endpoints | ✅ | ❌ none |

Two specifics worth naming:

**The port is thrown away.** `mesh/stun.go:44` returns `addr.Addr()`, and
`mesh_cmd.go:206` re-appends a hardcoded `51820`. The STUN query also uses an
*ephemeral socket, not the WireGuard bind* — the comment at `mesh/stun.go:36`
admits the port is "assumed to be port-preserved". That holds only on
port-preserving cone NAT. Every symmetric NAT falls to relay.

**LAN peers cannot find each other.** `mesh_cmd.go:186` unconditionally drops
RFC1918 candidates for privacy, with a comment claiming they are "exchanged
P2P/relay-brokered". **No such mechanism exists.** Two laptops on the same
Wi-Fi meet over the relay. This is a *dead claim*, which is worse than dead
code: a real filter relies on an imaginary fallback.

**IPv6 is dead weight.** `MeshIPv6` is assigned by Convex, persisted, and never
put on an interface (`mesh/manager.go:230` configures v4 only). Native IPv6 is
Tailscale's single biggest traversal win — it bypasses NAT entirely — and it is
absent.

### 3.3 Relay — one process, no mesh, silent drops

- **Both peers must be on the same relay process.** `relay/mesh.go:60` looks up
  `s.meshStreams[dst]` in per-process memory; a miss `continue`s — **silent
  drop, no error**. There is no relay↔relay forwarding, no region concept.
  DERP's entire multi-region topology has no counterpart.
- **>1 relay actively breaks mesh.** Each relay connection calls
  `globalMeshDERP.attach()`, which replaces a single global `t.stream` and
  closes the old one (`mesh_derp_transport.go:79`). N relays ⇒ last-attach-wins,
  and every reconnect flips which relay carries mesh.
- **Production is one relay.** `builtinRelayPins` has exactly one entry,
  `public.yaver.io` (`main.go:11415`). `relayCount: 1` in every client log.

### 3.4 Roaming — timers where Tailscale has events

There is **no network-change detection at all**: no netlink, no `NWPathMonitor`,
no route monitor. `mesh_agent.go:288` concedes it: *"Tailscale re-STUNs on every
link event; we do the coarser thing — re-register every 2 minutes."*

Worst case on a Wi-Fi→cellular switch: ~2 min advertising a dead endpoint, plus
20 s peer reconcile, plus 20 s DERP fallover, plus a 180 s `endpointRoamGrace`
during which reconcile refuses to overwrite the "live" dead endpoint. Tailscale
reacts sub-second. **PMTUD: none** — MTU is a fixed 1420 even on the relay path,
where the payload additionally rides QUIC-over-UDP.

## 4. Client & UI — where the product asserts things it does not know

Full ranked list in the appendix; these are the ones that caused this outage.

### 4.1 The self-inflicted relay ban

Stale password → every relay attempt 401s → five uncoordinated reconnect loops
retry → relay rate-limits the device (`too many invalid relay password
attempts`) → correct credentials are now refused. The credential refresh that
fixes it in 3 seconds (`[connect-relay-auth-refresh]`) exists but is not on the
storm path.

**Lesson: a retry loop that can trip an auth rate-limiter must refresh the
credential before its second attempt, and there must be exactly one loop.**

### 4.2 A permission denial reported as a routing verdict

`directProbeFailure.ts:79` classifies bare `"Network request failed"` as
`unroutable`. That string is RN's catch-all: it covers `ENETUNREACH`, TLS
resets, DNS failure, backgrounding — **and iOS Local Network permission
denial**. The app then tells the user:

> `unroutable (no route from this phone to that address right now — negative-cached until the network changes)`

The actual remedy is *Settings → Privacy → Local Network*. Nothing in
`mobile/src/` ever checks that permission (grep: zero matches). Granting it does
not change `deriveNetworkIdentity`, so the cache is not even wiped when the user
fixes it.

### 4.3 The negative cache could not see a tunnel

`deriveNetworkIdentity` (`quic.ts:49`) keys on Wi-Fi SSID and cellular carrier.
Bringing a VPN up changes neither, so `setNetworkIdentity` early-returns and
nothing is wiped — every `100.x` leg stays marked dead while the tailnet is up.
**Fixed 2026-07-20** (`unroutableCache.ts`: 45 s TTL for tunnel legs +
`forgetTunnelLegs()` at the top of every connect attempt).

Still open, same file: `observedTailnetUp` writes into `knownPathsUp`, which has
**no TTL and is not cleared on network flip** — negative evidence expires in
5 minutes, positive evidence is immortal. One successful tailnet reach ever ⇒
`Infinity` 100.x candidates raced forever, including after the VPN is disabled.

### 4.4 `unreachable` is accepted and dropped on the floor

`deviceStatus.ts:382` declares `unreachable?: boolean`; `:384` does not
destructure it. `devices.tsx:579` passes it in good faith. It has **zero
effect**, so a probe-proven-unreachable box still renders "ready to connect".
Web has exactly the guard mobile lacks (`web/lib/device-lifecycle.ts:240`).

**Meta-lesson, and the one worth internalising: the three highest-severity
client defects are all cases where the correct fix was designed, documented at
length in a comment, and then not wired up** — a dropped destructure, a map that
is never cleared, a setter with one caller. The comments assert behaviour the
code no longer has, so reading them actively misleads. *A comment describing an
invariant is not an invariant. Test it or delete it.*

## 5. The interop matrix — the actual requirement

Every Yaver device has mesh **on by default**. Users will be in every
combination. Mesh must be correct in all of them, and "correct" sometimes means
"stay out of the way".

| # | Phone | Box | Today | Required |
|---|---|---|---|---|
| 1 | no VPN | no VPN | Mesh may start; **no mobile data plane**, so LAN-guess or relay | Mesh direct, or relay with silent upgrade |
| 2 | Tailscale | Tailscale | **Mesh off on both.** Tailnet legs fail instantly (§4.2) | Detect tailnet, **prefer it**, do not duplicate it |
| 3 | Tailscale | no VPN | Mesh off on phone side; box may run mesh; ranges collide in classification | Mesh on box, tailnet unused for this pair, relay/direct |
| 4 | no VPN | Tailscale | Box mesh off; phone has no tailnet address | Relay, or mesh if box can be persuaded to run |
| 5 | other VPN (corp) | any | Untested. Guard only knows the CGNAT overlap | Detect *any* overlapping route, stand down per-route |
| 6 | Tailscale started **after** mesh | any | **Silent blackhole** of 100.96–100.111 peers | Re-evaluate on link events; withdraw the route |

The rule the matrix implies:

> **Yaver Mesh is a fallback overlay, not a competitor.** When a healthy
> incumbent tunnel already connects two Yaver devices, mesh must detect it,
> defer to it, and say so. Mesh should only build a path where none exists.

That is a strictly easier product than Tailscale, and it is the one users want.
It also makes case 2 — the dominant case — trivially correct instead of
catastrophically wrong.

## 6. What the literature says to build

Sources are primary where possible: Tailscale engineering posts, the RFC stack,
Apple developer docs. Claims that could not be verified are marked.

### 6.1 The one idea that changes everything: relay-first, upgrade silently

Tailscale's actual model, in their words:

> *"All connections start as relayed through a DERP server, and Tailscale then
> tries to upgrade them to a direct connection."*
> — [Connection types](https://tailscale.com/docs/reference/connection-types)
>
> *"Usually, after a few seconds, we'll have found a better path, and your
> connection transparently upgrades to it."*
> — [How NAT traversal works](https://tailscale.com/blog/how-nat-traversal-works)

Yaver does the exact opposite: it races LAN guesses **first**, treats the relay
as the failure branch, and only reaches it after the direct phase times out.
That is why a connect takes 20+ seconds and often ends in
`no path left to this device` — the logs in §1.2 are this inversion, verbatim.

Three properties fall out of relay-first, and together they *are* what "just
works" means to a user:

1. **Time-to-first-byte is relay RTT, not discovery latency.** The user never
   waits on NAT traversal.
2. **The upgrade is silent** — no reconnect, no session loss, no UI transition.
3. **Re-checking is continuous**, so a network change repairs itself.

**This is the highest-value change in this document and it requires no new
protocol.** It is a reordering of what Yaver already has.

Corollary, and a correction to how the product currently frames things:
measurements say **10–25% of device pairs will live on a relay permanently**
([RFC 5128 §4](https://www.rfc-editor.org/rfc/rfc5128.html) puts UDP hole
punching at >80%; [callstats/webrtcHacks](https://webrtchacks.com/usage-stats/),
billions of minutes, found **22% of sessions need a relay**). Tailscale's
self-reported ">90% direct" is higher because they add port-mapping and
birthday-paradox spraying — it is a vendor figure with no published methodology.
Either way: **the relay is a first-class datapath, not an error state.** One
relay for a permanent 10–25% of traffic is the capacity mistake underneath §3.3.

### 6.2 Do not allocate an overlay IP range — at all

This supersedes "relocate off 100.64/10" as the recommendation.

[RFC 6598 §4](https://www.rfc-editor.org/rfc/rfc6598.html) reserves
100.64.0.0/10 as Shared Address Space for CGN, and warns (§5.1) that
applications special-case it. Tailscale uses it because RFC 1918 collides with
home LANs constantly — and documents the resulting conflicts honestly, offering
two escapes: go IPv6-only, or constrain allocation with an
[IP pool](https://tailscale.com/kb/1304/ip-pool).

Neither escape is available to Yaver, because *Yaver is the guest.* You cannot
ask a user to disable IPv4 on their tailnet or edit their policy file because a
dev tool wants a subnet. And Tailscale's default allocator can hand any node an
address inside 100.96/12 at any time.

**Yaver's overlay is application-layer. It has no packet-forwarding plane, so it
does not need addresses at all.** It already has a better namespace: `deviceId`
and user aliases. Route by identity. Every address the overlay does not invent
is an entire class of collision bug that cannot happen — including the
start-order blackhole in §3.1.

If a routable address is ever genuinely required, the standards-clean answer is
**IPv6 ULA ([RFC 4193](https://www.rfc-editor.org/rfc/rfc4193.html))
`fd00::/8` with a per-user random /40**, which has a documented collision
probability and is the mechanism designed for exactly this. Never a hardcoded
CGNAT subrange.

### 6.3 QUIC is the right substrate — verify one setting

Yaver already runs QUIC. That is a **better** starting point than
WireGuard-over-UDP for an app-layer overlay, because
[RFC 9000 §9](https://www.rfc-editor.org/rfc/rfc9000.html#section-9) gives
connection migration natively:

- §9.1 probe a new path **without committing the connection to it** — this is
  structurally the same operation as a disco ping, already in the protocol.
- `PATH_CHALLENGE`/`PATH_RESPONSE` prove bidirectional reachability.
- §9.3 anti-spoofing and amplification limits are specified, not improvised.
- §9.4 congestion state **must** reset on path change.
- §9.5 use a different connection ID per path or an observer correlates the user.

**Action with a hard dependency: zero-length connection IDs silently foreclose
migration entirely.** Migration requires non-zero-length CIDs so the receiver
can distinguish a path change from a spoof. If Yaver's QUIC config uses
zero-length CIDs, nothing else in this section will fix the wifi→cellular story.
**Check this first — it is a ten-minute check with an outsized consequence.**

Borrow from WireGuard the *semantics*, not the datapath: an authenticated packet
from a new source address rehomes the peer
([whitepaper §2.1](https://www.wireguard.com/papers/wireguard.pdf)), and the
calibrated timers (10 s keepalive-timeout, 180 s reject-after-time).
**Multipath QUIC is still `draft-ietf-quic-multipath-21`, not an RFC — do not
build on it.**

### 6.4 Minimal disco, not full ICE

Full [RFC 8445](https://www.rfc-editor.org/rfc/rfc8445.html) ICE brings SDP,
offer/answer, agent roles and the pair-priority formula. Yaver needs ICE's
*idea* — gather candidates, race them in parallel, keep the best — not its
machinery. Tailscale reached its numbers with a purpose-built disco, not ICE.

Two specifics that fix named defects in §3.2:

- **STUN must be sent from the same socket that carries data.** Each socket gets
  its own NAT mapping, which is precisely why `mesh/stun.go`'s ephemeral-socket
  query returns a mapping WireGuard never uses. The taxonomy that matters is
  Endpoint-Independent Mapping ("easy") vs Endpoint-Dependent ("hard"), not the
  old cone/symmetric vocabulary — and note
  [RFC 8489](https://www.rfc-editor.org/rfc/rfc8489.html) *removed* RFC 3489's
  NAT-type discovery because classification by probing is unreliable.
- **Exchange candidate bundles over the relay, not Convex.** This resolves the
  privacy conflict cleanly: candidate endpoints are LAN IPs, which the project's
  Convex contract forbids. Peer-to-peer over the existing relay datapath is both
  privacy-preserving and what Tailscale's `CallMeMaybe` actually is. Convex
  stays identity + relay-map only.

Add **`trustBestAddrUntil`** damping: a challenger path must beat the incumbent
by a margin to displace it. Without it a path racer oscillates, and flapping is
worse than a stable relay. Stagger parallel attempts using
[RFC 8305](https://www.rfc-editor.org/rfc/rfc8305.html) §5 Happy Eyeballs v2
constants (250 ms default, 100 ms min, 2 s max) rather than inventing delays.

### 6.5 iOS — the current app-layer choice is correct; keep it

The audit's app-layer design is **validated** by the constraints, not compromised
by them. A real VPN would need `NEPacketTunnelProvider`, which brings:

- a **~15 MB memory ceiling** — Tailscale documented that this blocked them from
  shipping HTTP/2 and UPnP on iOS ([Hey linker, can you spare a
  meg?](https://tailscale.com/blog/go-linker)); reportedly raised since, no
  current Apple figure found *(unverified)*,
- an entitlement that must be **requested from Apple**,
- occupation of **the single VPN slot** — already forbidden by project policy,
- suspension shortly after screen-off *(community report, not Apple-primary)*.

What app-layer gives up is holding a connection open while backgrounded. That is
the real constraint, and it argues for **sub-second relay reconnect on
foreground** over heroic direct-path preservation.

On the permission itself, confirming §4.2 from Apple's side: local-network
checks are enforced "at the lowest levels of the system and thus apply to **all**
networking APIs, including Network framework, BSD Sockets, NSStream, and
NSURLSession" ([FAQ-2](https://developer.apple.com/forums/thread/663874)) — so
raw sockets do **not** dodge it. Cellular and VPN interfaces are *excluded* from
the local-network definition
([TN3179](https://developer.apple.com/documentation/technotes/tn3179-understanding-local-network-privacy)),
which is a useful discriminator: a tailnet address is not local-network-gated,
so a tailnet leg failing instantly is **not** the permission — it is ATS or no
route, and the two must be distinguished.

There is **no API to read the permission state without triggering the prompt**;
the `NWBrowser`+`NWListener` probe pair is the accepted workaround. Two
sequencing rules follow: get the user connected over the relay *first*, then ask
for LAN permission with context; and **never probe `.local` on iOS on the
connect path** — it needs the same permission and hangs rather than failing.

*Unverified and worth ten minutes on-device:* whether `Network.framework` and
BSD sockets are exempt from **ATS** (strong secondary consensus, no primary
Apple statement located). If they are, ATS constrains only Yaver's HTTP legs and
`NSAllowsArbitraryLoads` may be unnecessary.

### 6.6 The minimum viable path discovery

```
connect(peer):
  1. open QUIC to home relay        → session usable at t ≈ 1 RTT   [ALWAYS]
  2. fetch peer candidate bundle over the relay (never via Convex)
  3. probe ALL candidates in parallel, staggered ~250 ms (RFC 8305):
       - local interface addrs   (skip on iOS unless LN permission granted)
       - STUN reflexive addr     (from the DATA socket)
       - port-mapped addr        (UPnP-IGD / NAT-PMP / PCP)
     authenticated ping carrying a TxID for RTT matching
  4. first PATH_RESPONSE from a direct candidate:
       validate (RFC 9000 §9.1) → migrate → reset CC (§9.4)   [silent]
  5. hold the trust window; re-probe every 20-26 s; on link event, restart at 2
  6. any path failure → fall back to relay immediately, NEVER to "disconnected"
```

**Measure before optimising.** Every number above comes from someone else's
network population, and Tailscale's >90% and callstats' 22% differ ~4× because
the populations differ. Instrument what fraction of Yaver pairs actually end up
on the relay, and how long each leg takes. That single measurement decides
whether port-mapping or multi-region relay is worth more.

## 7. Remediation, ordered by (user pain ÷ effort)

**P0 — stop the self-harm. No new architecture required.**

1. **One reconnect loop per device, globally.** Five concurrent loops against
   one relay host is what trips the ban and exhausts iOS's ~6-socket-per-host
   budget.
2. **Refresh relay credentials before the second attempt**, not after N
   failures. The 3-second self-heal already exists; move it onto the storm path.
3. **Back off *hard* on a rate-limit reply** and surface it as its own state.
   `too many invalid relay password attempts` must never be retried on the same
   schedule as a timeout.
4. **Detect and surface iOS Local Network permission.** A denial must read
   *"Yaver needs Local Network permission — Settings → Privacy"*, must **not**
   be negative-cached, and granting it must wipe the cache.
5. **Expire `knownPathsUp`** and clear it on network flip (§4.3).
6. **Wire `unreachable`** in `deviceStatus.ts`; adopt web's `contradicted` gate;
   align `HEARTBEAT_STALE_MS` (mobile 900 s vs shared 90 s).
7. **Age out stale candidates.** A Docker bridge and a months-old hotspot range
   should not be raced. Cap candidates and drop addresses not seen in N
   heartbeats.

**P1 — make the relay survivable.**

8. **More than one relay, and make >1 actually work** — fix the single
   `t.stream` (`mesh_derp_transport.go:79`) before adding capacity, or a second
   relay makes things *worse*.
9. **Relay↔relay forwarding**, or at minimum pin both peers of a pair to the
   same relay deterministically instead of by race.
10. **Never silently drop a mesh frame** (`relay/mesh.go:60`).

**P2 — earn the direct path.**

11. **Use the STUN port, from the WireGuard socket.** This alone converts a
    large fraction of symmetric-NAT pairs from relay to direct.
12. **Multiple candidates per peer + connectivity checks.** WireGuard accepts
    one endpoint, so probe several and *set* the winner — that is ICE's core
    idea and it does not require adopting ICE wholesale.
13. **Publish LAN candidates over the P2P/relay channel** the privacy filter
    already assumes exists, or delete the claim.
14. **IPv6 endpoints.** Highest traversal value per line of code.
15. **Link-event triggered re-STUN** (`NWPathMonitor` / netlink) instead of the
    2-minute timer.

**P3 — the range. Stop having one.**

16. **Do not relocate the range — delete it** (§6.2). An application-layer
    overlay with no forwarding plane does not need addresses; route by
    `deviceId`. This is strictly better than moving to another range, because it
    removes the collision *class* rather than the current instance, and it is
    the only option that stays correct when the next VPN the user installs picks
    an overlapping range.
17. Until that lands: re-run the conflict guard **on link events**, not only at
    bring-up, and **withdraw** the mesh route when a conflicting one appears —
    that is the start-order blackhole in §3.1.

## 8. What would have told us this in ten seconds

Every fault above was invisible because the system reported success shapes.

- `yaver mesh status` said *"has not completed yet — retries on a backoff"* on a
  box where the honest answer was *"will never start while Tailscale is up"*.
  **A backoff message must never stand in for a permanent refusal.**
- The connect log said `unroutable — no route from this phone to that address`
  for a socket the OS refused locally. **Report the failure you observed, not
  the one you inferred.** A 0 ms failure and a 2 s failure are different events
  and must never share a label.
- `[connect-resume] Already connected` printed 9 s after `no path left to this
  device`. **A cached success must expire faster than the thing it describes.**
- `Racing N candidate(s)` omits negative-cached legs — so the diagnostic users
  are told to read hides the leg that matters. **A diagnostic that filters its
  own input is not a diagnostic.**

The probe that would have caught all of it in one line, and which should exist:

```
yaver net explain <device>
  mesh        : OFF — 100.96/12 conflicts with utun4 (Tailscale 100.64/10)
  tailnet     : phone 100.69.93.34 ↔ box 100.89.155.25 — REACHABLE (12 ms)
  lan         : 5 candidates, 0 answered (Local Network permission: DENIED)
  relay       : public-free — RATE-LIMITED until 09:24 (invalid password ×7)
  verdict     : tailnet is up and usable; the app is not using it
```

## 9. Relay tiers × Mesh × the three topologies (2026-07-20)

Two facts first, both verified in code, because the plan depends on them:

- **There is no Free/Pro relay tier.** `grep` for a relay tier model returns
  nothing; `"pro"` in this repo is a *cloud VM plan* (`cloud_deploy.go:115`).
  The relay has free-tier *quota/ownership* validation (`relay/server.go:452`)
  but one hardcoded host (`builtinRelayPins`) and no paid lane.
- **Mesh deferral is host-global, not per-peer.** `SubnetRouteConflict`
  (`mesh/subnet_conflict.go:25`) enumerates every interface and, on any hit,
  mesh refuses to `Start()` **at all**.

### 9.1 The hybrid case is the one that exposes the design error

| Topology | Mini | Phone | Today | Should be |
|---|---|---|---|---|
| **A. Neither on a VPN** | mesh may start | no mobile mesh data plane | relay only | mesh direct, relay while it forms |
| **B. Both on Tailscale** | mesh defers | tailnet works | tailnet (if legs not miscached) | **defer — correct** |
| **C. Hybrid** — mini on Tailscale, phone not | **mesh refuses to start** | not on tailnet | **relay only** | **mesh SHOULD serve this pair** |

Case C is the important one, and it is currently wrong for a precise reason:

> Tailscale **cannot carry the phone↔mini pair at all** — the phone is not on
> the tailnet. Yet mesh stands down on the mini because of a conflict that only
> concerns *tailnet peers*. Mesh defers to an incumbent that cannot serve the
> very peer it is deferring about.

The deferral question is **per-peer** — "can the incumbent reach *this* peer?" —
but the code asks it **per-host**: "does a conflicting interface exist anywhere?"
A host-global answer to a per-peer question is why every Yaver device on a
Tailscale box loses mesh for *all* peers, including the ones Tailscale never had.

### 9.2 The fix: /32 host routes, not a /12 claim

The conflict exists only because mesh claims the whole `100.96.0.0/12`, which
collides with Tailscale's single `100.64.0.0/10` route. It does not need to.

**Install a `/32` host route per mesh peer instead of one `/12` blanket route.**

- A `/32` is the longest possible prefix, so it wins for exactly that address
  and touches nothing else — no fight with Tailscale's `/10`.
- Install a `/32` **only** for peers the incumbent cannot reach. If Tailscale
  already carries a peer, do not route it: that is deferral, correctly scoped.
- Case C resolves: the mini installs a `/32` for the phone (which Tailscale
  cannot reach) and no route for tailnet peers. Both overlays coexist on the
  same host, each serving the peers it can.
- The start-order blackhole (§3.1) disappears too: mesh never owns a range wide
  enough to swallow a Tailscale address it doesn't know about.

This is strictly less invasive than today and it is the intermediate step
toward §6.2 (allocate no range at all). If mesh keeps addresses, per-peer `/32`
is how it stops being a competitor for the routing table.

### 9.3 What the relay tiers should be

The relay is the **only** path that works in every topology, and measurements
say 10–25% of pairs will live there permanently (§6.1). It is therefore product
surface, not failure surface — and today it is one box, `relayCount: 1`, which
the client can get itself banned from (§4.1).

**Yaver Free Relay** — the default, and it must be *correct*, not merely
present:
- multi-region with latency-based home selection (§6.1 P1), because one host in
  fsn1 is a latency floor for every non-EU user;
- fair-use quota that **degrades honestly** — a throttled relay must say
  "throttled, upgrade or wait", never time out and let the client conclude
  "no path left to this device";
- the rate-limit fix from §7 P0-3: a ban is a distinct state, not a timeout.

**Yaver Relay Pro** — the paid lane. What legitimately belongs behind it is
*capacity and placement*, never *connectivity*:
- dedicated/regional egress, higher or no throughput cap, priority under load;
- self-hostable relay with the same protocol (already supported — keep it);
- **not** "your devices can connect at all". Connectivity is the product's
  promise; metering it turns an outage into a sales page, and the free tier is
  what makes direct-path failures survivable for everyone else.

Ordering note: **fix the client's self-inflicted ban (§7 P0-1..3) before adding
relay capacity.** Today more relays make things *worse* — `mesh_derp_transport.go:79`
keeps a single global `t.stream`, so N relays means last-attach-wins and every
reconnect flips which relay carries mesh.

### 9.4 The rule, stated once

> **Relay is the floor; mesh is an optimisation; the incumbent VPN is a peer,
> not a rival.** Every pair gets a working path immediately via the relay, then
> upgrades to the best available: incumbent tunnel if it reaches, else mesh
> direct, else stay. Deferral is decided per peer, never per host.

Case A gets mesh. Case B gets Tailscale. Case C gets mesh for the pair Tailscale
cannot serve and Tailscale for the pairs it can — on the same machine, at the
same time.
